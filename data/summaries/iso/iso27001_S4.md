# Support & Resources

## Support (Clause 7)

Clause 7 addresses the **resources and support** needed to establish, implement, maintain, and improve the ISMS.

## Resources (Clause 7.1)

Organizations must determine and provide resources needed for the ISMS.

### Resource Categories

| Category | Examples |
|----------|----------|
| **Human** | Security team, IT staff, management time |
| **Financial** | Budget for tools, training, consultants |
| **Technical** | Security tools, infrastructure, systems |
| **Information** | Threat intelligence, standards, guidance |
| **Time** | Adequate schedule for implementation |

### Resource Planning

Consider resources for:
- Initial ISMS implementation
- Ongoing operations
- Continual improvement
- Incident response
- Audit and certification

---

## Competence (Clause 7.2)

Organizations must ensure that personnel have the **competence** needed for their roles.

### Competence Requirements

| Step | Action |
|------|--------|
| **Determine** | Identify competence needed for ISMS roles |
| **Ensure** | Verify personnel have required competence |
| **Take action** | Provide training or acquire competence |
| **Evaluate** | Assess effectiveness of actions |
| **Retain evidence** | Document competence |

### Competence Areas

| Role | Competence Needed |
|------|-------------------|
| **ISMS Manager** | Risk management, standards, project management |
| **Security Analyst** | Threat analysis, incident response, tools |
| **IT Administrator** | System hardening, access control, patching |
| **Auditor** | Audit techniques, standards, reporting |
| **All Staff** | Security awareness, policy compliance |

### Acquiring Competence

Options include:
- Training courses
- Certifications (CISSP, CISM, ISO 27001 Lead Implementer)
- Mentoring and coaching
- On-the-job experience
- Hiring qualified personnel
- Outsourcing to specialists

---

## Awareness (Clause 7.3)

Personnel must be **aware** of:

| Awareness Area | Content |
|----------------|---------|
| **Policy** | The information security policy |
| **Contribution** | Their contribution to ISMS effectiveness |
| **Benefits** | Benefits of improved security performance |
| **Consequences** | Implications of not conforming |

### Awareness Program Elements

| Element | Description |
|---------|-------------|
| **Induction** | Security briefing for new starters |
| **Regular updates** | Ongoing awareness communications |
| **Campaigns** | Focused awareness initiatives |
| **Simulations** | Phishing tests, social engineering exercises |
| **Reminders** | Posters, screensavers, newsletters |
| **Metrics** | Measure awareness effectiveness |

## Communication (Clause 7.4)

Organizations must determine communication needs.

### Communication Planning

| Question | Consideration |
|----------|---------------|
| **What** | What needs to be communicated? |
| **When** | When should communication occur? |
| **Who (audience)** | Who needs to receive it? |
| **Who (sender)** | Who should communicate? |
| **How** | What channels and methods? |

### Communication Types

| Type | Examples |
|------|----------|
| **Internal** | Policy updates, incident alerts, training |
| **External** | Customer notifications, regulatory reporting |
| **Routine** | Regular updates, status reports |
| **Event-driven** | Incident notifications, change announcements |

## Documented Information (Clause 7.5)

### Required Documentation

The ISMS must include documented information:
- Required by ISO 27001
- Determined necessary for ISMS effectiveness

### Creating and Updating (7.5.2)

When creating or updating documents, ensure:
- **Identification** — Title, date, author, version
- **Format** — Appropriate format and media
- **Review and approval** — Authorized before use

### Control of Documented Information (7.5.3)

Documents must be controlled to ensure:
- **Availability** — Accessible when needed
- **Protection** — Secure from loss, misuse, unauthorized access

Control activities include:
- Distribution and access control
- Storage and preservation
- Change control
- Retention and disposition

### Document Types

| Type | Purpose | Examples |
|------|---------|----------|
| **Policies** | State management intent | Security policy, acceptable use |
| **Procedures** | Describe how to perform activities | Incident response, access provisioning |
| **Standards** | Specify requirements | Password standards, encryption standards |
| **Guidelines** | Provide recommendations | Secure coding guidelines |
| **Records** | Provide evidence | Audit reports, risk assessments |

## Annex A: Asset Management (A.5.9–A.5.14)

### A.5.9 Inventory of Information and Other Assets

**Control:** An inventory of information and associated assets shall be developed and maintained.

**Inventory should include:**
- Asset identification
- Owner
- Location
- Classification
- Value/criticality

### A.5.10 Acceptable Use of Information and Assets

**Control:** Rules for acceptable use shall be identified, documented, and implemented.

**Cover:**
- Permitted uses
- Prohibited uses
- Personal use policies
- Monitoring disclosure

### A.5.11 Return of Assets

**Control:** Personnel shall return assets upon termination or change.

### A.5.12 Classification of Information

**Control:** Information shall be classified according to needs.

**Common classification levels:**
- Public
- Internal
- Confidential
- Restricted/Secret

### A.5.13 Labelling of Information

**Control:** Procedures for labelling shall be developed.

### A.5.14 Information Transfer

**Control:** Rules and procedures for information transfer shall be in place.

---

## Annex A: Supplier Relationships (A.5.19–A.5.22)

### A.5.19 Information Security in Supplier Relationships

**Control:** Processes shall be defined to manage security risks from suppliers.

### A.5.20 Addressing Security in Supplier Agreements

**Control:** Security requirements shall be established in agreements.

**Include:**
- Security requirements
- Right to audit
- Incident notification
- Data handling
- Termination provisions

### A.5.21 Managing Security in the ICT Supply Chain

**Control:** Processes shall address ICT supply chain risks.

### A.5.22 Monitoring, Review and Change Management of Supplier Services

**Control:** Supplier security shall be monitored and reviewed.

## Key Takeaways

1. **Resources enable success** — Adequate investment is essential
2. **Competence matters** — Right skills in right roles
3. **Awareness is ongoing** — Not a one-time event
4. **Communication is planned** — Know what, when, who, how
5. **Documentation provides evidence** — Controlled and current
6. **Assets must be managed** — Know what you have, protect it
